How to Make Changes to this Website

What this page is for

This guide explains the practical, real-world steps to maintain the FFJ Consulting LLC website: where the code lives on your laptop, how the site is hosted on AWS, how to update content (especially the resume), how to deploy, and how to push updates to GitHub.

Quick Summary (Most Common Workflow)

  1. Edit the content (resume, articles, links, docs) in the repo
  2. Commit locally and push to GitHub
  3. Deploy to AWS (S3 + CloudFront invalidation)
  4. Hard refresh / use incognito to confirm the update (cache)

How the website is hosted on AWS

This website is a static React build hosted on AWS. The key AWS resources involved are:

  • S3 bucket (static site files): ffj-consulting-website
  • CloudFront distribution (CDN + HTTPS): E3545N3N8YO2FZ
  • ACM certificate (TLS/HTTPS): issued for ffjconsultingllc.com and/or www.ffjconsultingllc.com
  • Route 53 (DNS): routes the domain to CloudFront

Important: Caching

CloudFront caches files. Even after a successful deploy, your browser may still show old content. If something “didn’t update,” try:

  • Wait 1–5 minutes (CloudFront invalidation)
  • Hard refresh (Cmd+Shift+R)
  • Use incognito / private window

Technical architecture (high level)

Browser
  -> CloudFront (CDN + TLS)
      -> S3 bucket (static files: index.html, assets/, content/, docs/, images/)

There is also a backend project in this repository (ASP.NET Core), but the current production site is primarily a static frontend hosted from S3/CloudFront. The backend can be deployed later if you choose.

Where content lives in the repo

  • React app: frontend/src
  • Static content served at runtime: frontend/public
  • Docs pages served at /docs: frontend/public/docs (copied to dist/docs when building)
  • Resume served at /content/resume.md: frontend/public/content/resume.md

Docs folder note

There are two copies of docs in this repo: Docs/ and frontend/public/docs/. The deployed website uses frontend/public/docs/. Keep them in sync (or treat frontend/public/docs/ as the source of truth).

How to run locally (preview changes before deploy)

Then open the URL Vite prints (typically http://localhost:3000 or http://localhost:5173).

How to deploy to AWS (recommended)

Use the automated deploy script (builds the frontend, syncs to S3, fixes JS MIME type, invalidates CloudFront):

cd /Users/fjabbari/REPO_CURSOR/FFJ-CONSULTING-LLC
./deploy-with-new-domain.sh

This deploy targets: ffj-consulting-website (S3), CloudFront distribution E3545N3N8YO2FZ, AWS profile my-sso, region us-east-1.

How to update the website resume (markdown)

The Resume page reads /content/resume.md at runtime, so update:

  • frontend/public/content/resume.md (this is what gets deployed)
  • frontend/src/content/resume.md (keep in sync for source editing/history)

After updating, deploy:

Common troubleshooting

“My change deployed but I don’t see it”

  1. Open the raw file directly (example): https://ffjconsultingllc.com/content/resume.md
  2. Use incognito/private window
  3. Hard refresh (Cmd+Shift+R)
  4. Wait for CloudFront invalidation to complete

“Links I add on the AI pages don’t show for everyone”

The “Add Link” feature currently stores new entries in your browser (localStorage) so you can quickly curate links without a backend. That means it is per-browser/per-device. If you want a shared admin UI for everyone, that requires an AWS backend (API + database).
